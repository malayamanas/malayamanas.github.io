Short Description An adversary targets credentials or authentication material to enable further compromise through SAM Registry Hive Handle Requested. Sysmon Event(s) 1, 10 (Process Creation, ProcessAccess) provide the primary Sysmon-layer detection signal for this technique.

2. MITRE ATT&CK Mapping

  • Tactic : Credential Access

Technique : OS Credential Dumping: Security Account Manager (T1003.002)

3. How the Attack Typically Works (Procedure)

  1. Attacker achieves code execution on the target system with sufficient privileges to write to the target registry key.
  2. Sysmon Event 13 (RegistryValueSet) fires when the persistence registry value is written — this is the primary Sysmon detection signal, capturing the key path, data, and writing process.
  3. For key creation (new Run entry or IFEO key), Sysmon Event 12 (RegistryCreateDelete) fires first.
  4. On trigger (logon, process start, system boot), the registered payload executes — Sysmon Event 1 captures the spawned process with the persistence path as Image.
  5. Sysmon Event 7 (ImageLoaded) fires when DLL-based persistence (AppInit_DLLs, Security Packages) loads the malicious DLL into target processes.
Most common tools observed in 2024–2026:
  • Reg.exe / PowerShell Set-ItemProperty
  • CobaltStrike registry persistence
  • Metasploit persistence modules
  • SharPersist

4. Key Sysmon Event Indicators

Sysmon Event 1 — Process Creation

A new process was created. Captures image path, command line, parent, hashes, user, and integrity level.

Sysmon Event 10 — ProcessAccess

A process opened a handle to another process. Captures source/target process and the access rights (GrantedAccess) requested.

FieldSuspicious Value / PatternWhy It Matters
TargetImagelsass.exe (credential dumping), winlogon.exe (credential interception), csrss.exe (system process injection), ntoskrnl.exe (kernel access)Process being accessed; lsass.exe is the primary target for credential dumping — any unexpected process access is high-priority
GrantedAccess0x1010 (PROCESS_VM_READ + PROCESS_QUERY_INFORMATION — classic LSASS dump), 0x1410 (adds PROCESS_VM_OPERATION), 0x1F0FFF (full access), 0x40 (DuplicateHandle for token theft)Access rights granted to the handle; specific bit patterns reveal the exact operation being performed (memory read, token theft, etc.)

5. Detection / Hunting Recommendations

Strong / Classic Signature Rule (Critical Confidence)
  • Alert on Sysmon event(s) 1, 10 matching the field combinations shown in Section 4
  • Apply environment-specific thresholds: a single matching event may be sufficient for critical-level detections
Sigma Detection Rule (YAML skeleton): 
title: SAM Registry Hive Handle Requested
status: experimental
logsource:
    category: process_creation  # adjust per primary Sysmon event type
    product: windows
detection:
    selection:
    EventID: 1  # Sysmon Process Creation
    ParentImage|endswith|any:  # Suspicious parent
      - '\\winword.exe'
      - '\\excel.exe'
      - '\\powerpnt.exe'
      - '\\w3wp.exe'
    Image|endswith|any:
      - '\\cmd.exe'
      - '\\powershell.exe'
      - '\\wscript.exe'
    condition: selection
level: critical
tags:
    - collection
    - credential_access
    - t1003.002
Contextual / Behavioral Hunting Queries (SIEM / EDR)
  • Hunt for Sysmon 1 / Sysmon 10 events where Image is NOT in the approved software baseline
  • Alert on Sysmon 1 / Sysmon 10 from accounts in privileged groups outside approved maintenance windows
  • Correlate Sysmon 1 events with subsequent Windows Security 4624/4625 events to link process activity to authentication events
  • Stack-rank Sysmon 1 / Sysmon 10 events by Image path to identify rare/novel executables across the fleet
False Positive Considerations
  • Legitimate software installers write to Run keys on installation
  • Enterprise management platforms (SCCM, Intune) deploy registry configurations
  • AV/EDR agents modify their own registry configuration at startup
  • Group Policy preferences apply registry settings across domain members
Recommended Hardening (Prevention & Impact Reduction)
  • Implement attack surface reduction (ASR) rules to block persistence via Run keys and AppInit_DLLs
  • Use application allowlisting (WDAC/AppLocker) to restrict which processes can modify persistence-relevant registry paths
  • Enable Sysmon registry monitoring for all persistence-relevant key paths (RunOnce, AppInit, IFEO, LSA)
  • Restrict registry write permissions on HKLM hives to SYSTEM and TrustedInstaller only

6. Recommended Response Actions

  • Isolate the affected host from the network immediately if active exploitation or lateral movement is confirmed.
  • Identify the initiating account (from Sysmon Event 1 User field or Windows Security 4624 correlation) and disable it pending investigation.
  • Preserve Sysmon event logs from the affected host(s) before any remediation — export the Microsoft-Windows-Sysmon/Operational channel.
  • Correlate the Sysmon events with Windows Security logs using the LogonId field to link process activity to authenticated sessions.
  • Identify and remove the persistence mechanism (registry key, scheduled task, service, WMI subscription) from all affected hosts.
  • Search for the same persistence IOC (file hash, registry key path and value, task name) across all endpoints in the fleet.
  • Review the affected account's full Sysmon event history for the past 30 days to understand the full scope of attacker activity.
  • Rebuild the affected host from a known-good image if multiple persistence mechanisms or deep kernel-level persistence is suspected.
