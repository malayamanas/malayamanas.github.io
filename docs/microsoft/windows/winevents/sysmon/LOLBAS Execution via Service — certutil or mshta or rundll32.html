Short Description An adversary runs attacker-controlled code on the target system through LOLBAS Execution via Service — certutil or mshta or rundll32. Sysmon Event(s) 1, 3, 7 (Process Creation, Network Connection, Image Loaded) provide the primary Sysmon-layer detection signal for this technique.

3. How the Attack Typically Works (Procedure)

  1. Attacker achieves initial access to the target environment through social engineering, exploitation, or stolen credentials.
  2. Using the established foothold, the attacker performs runs attacker-controlled code on the target system related to LOLBAS Execution via Service — certutil or mshta or rundll32.
  3. MITRE ATT&CK technique T1218 is leveraged using tools and techniques observed consistently in 2024–2026 campaigns.
  4. Sysmon Event(s) 1, 3, 7 are generated, capturing critical process, network, file, or registry activity depending on the specific technique.
  5. Post-attack, the attacker may perform additional lateral movement, persistence, or data collection based on the access and capabilities gained.
Most common tools observed in 2024–2026:
  • CobaltStrike (commercial C2 framework)
  • Metasploit Framework (open-source exploitation)
  • Impacket (Python network protocol library)
  • CrackMapExec (post-exploitation framework)
  • PowerSploit / PowerView (PowerShell attack modules)

5. Detection / Hunting Recommendations

Strong / Classic Signature Rule (High Confidence)
  • Alert on Sysmon event(s) 1, 3, 7 matching the field combinations shown in Section 4
  • Apply environment-specific thresholds: a single matching event may be sufficient for critical-level detections
Sigma Detection Rule (YAML skeleton): 
title: LOLBAS Execution via Service — certutil or mshta or rundll32
status: experimental
logsource:
    category: process_creation  # adjust per primary Sysmon event type
    product: windows
detection:
    selection:
    EventID: 1  # Sysmon Process Creation
    Image|endswith: '\\certutil.exe'
    CommandLine|contains|any:
      - '-urlcache'
      - '-decode'
      - '-encode'
      - 'http://'
      - 'https://'
    condition: selection
level: high
tags:
    - defense_evasion
    - execution
    - t1218
    - t1569.002
Contextual / Behavioral Hunting Queries (SIEM / EDR)
  • Hunt for Sysmon 1 / Sysmon 3 / Sysmon 7 events where Image is NOT in the approved software baseline
  • Alert on Sysmon 1 / Sysmon 3 / Sysmon 7 from accounts in privileged groups outside approved maintenance windows
  • Correlate Sysmon 1 events with subsequent Windows Security 4624/4625 events to link process activity to authentication events
  • Stack-rank Sysmon 1 / Sysmon 3 / Sysmon 7 events by Image path to identify rare/novel executables across the fleet
False Positive Considerations
  • Legitimate software update mechanisms use these tools for downloads
  • Enterprise asset management and software packaging use certutil for certificate validation
  • Custom in-house scripts may use WScript/CScript for automation
  • DNS queries from legitimate browsers and update services are high volume
Recommended Hardening (Prevention & Impact Reduction)
  • Apply Principle of Least Privilege: restrict the ability to perform this attack to the minimum required accounts
  • Deploy comprehensive Sysmon configuration (SwiftOnSecurity or Florian Roth config) to capture all relevant events
  • Configure SIEM correlation rules to baseline normal Sysmon patterns and alert on deviations
  • Enable Microsoft Defender for Endpoint or equivalent EDR for behavioral detection in addition to Sysmon event-based detection

6. Recommended Response Actions

  • Isolate the affected host from the network immediately if active exploitation or lateral movement is confirmed.
  • Identify the initiating account (from Sysmon Event 1 User field or Windows Security 4624 correlation) and disable it pending investigation.
  • Preserve Sysmon event logs from the affected host(s) before any remediation — export the Microsoft-Windows-Sysmon/Operational channel.
  • Correlate the Sysmon events with Windows Security logs using the LogonId field to link process activity to authenticated sessions.
  • Identify and remove the persistence mechanism (registry key, scheduled task, service, WMI subscription) from all affected hosts.
  • Search for the same persistence IOC (file hash, registry key path and value, task name) across all endpoints in the fleet.
  • Review the affected account's full Sysmon event history for the past 30 days to understand the full scope of attacker activity.
  • Rebuild the affected host from a known-good image if multiple persistence mechanisms or deep kernel-level persistence is suspected.
