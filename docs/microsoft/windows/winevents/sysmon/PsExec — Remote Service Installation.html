Short Description An adversary runs attacker-controlled code on the target system through PsExec — Remote Service Installation. Sysmon Event(s) 1, 3, 17, 18 (Process Creation, Network Connection, PipeEvent (Pipe Created), PipeEvent (Pipe Connected)) provide the primary Sysmon-layer detection signal for this technique.

2. MITRE ATT&CK Mapping

  • Tactic : Execution

Technique : System Services: Service Execution (T1569.002)

3. How the Attack Typically Works (Procedure)

  1. Attacker obtains domain or local administrator credentials through credential dumping or lateral movement.
  2. PsExec or equivalent tool connects to the target host's ADMIN$ share (Sysmon Event 3 — TCP 445), copies a service binary (PSEXESVC.exe or renamed variant), and creates it as a Windows service.
  3. Sysmon Event 11 (FileCreate) captures the binary drop on ADMIN$; Sysmon Event 17/18 captures the named pipe communication channel (\PSEXESVC or \RemCom).
  4. The service executes the attacker's command under SYSTEM context — Sysmon Event 1 captures the spawned process with parent PSEXESVC.exe.
  5. PsExec removes the service binary after execution, but Sysmon Events 1, 3, 11, 17, 18 collectively capture the full activity chain.
Most common tools observed in 2024–2026:
  • Sysinternals PsExec
  • Impacket psexec.py (RemCom pipe)
  • CobaltStrike psexec jump
  • Metasploit psexec_psh module

5. Detection / Hunting Recommendations

Strong / Classic Signature Rule (High Confidence)
  • Alert on Sysmon event(s) 1, 3, 17, 18 matching the field combinations shown in Section 4
  • Apply environment-specific thresholds: a single matching event may be sufficient for critical-level detections
Sigma Detection Rule (YAML skeleton): 
title: PsExec — Remote Service Installation
status: experimental
logsource:
    category: process_creation  # adjust per primary Sysmon event type
    product: windows
detection:
    selection:
    EventID: 1  # Sysmon Process Creation
    ParentImage|endswith|any:  # Suspicious parent
      - '\\winword.exe'
      - '\\excel.exe'
      - '\\powerpnt.exe'
      - '\\w3wp.exe'
    Image|endswith|any:
      - '\\cmd.exe'
      - '\\powershell.exe'
      - '\\wscript.exe'
    condition: selection
level: high
tags:
    - lateral_movement
    - execution
    - t1569.002
Contextual / Behavioral Hunting Queries (SIEM / EDR)
  • Hunt for Sysmon 1 / Sysmon 3 / Sysmon 17 / Sysmon 18 events where Image is NOT in the approved software baseline
  • Alert on Sysmon 1 / Sysmon 3 / Sysmon 17 / Sysmon 18 from accounts in privileged groups outside approved maintenance windows
  • Correlate Sysmon 1 events with subsequent Windows Security 4624/4625 events to link process activity to authentication events
  • Stack-rank Sysmon 1 / Sysmon 3 / Sysmon 17 / Sysmon 18 events by Image path to identify rare/novel executables across the fleet
False Positive Considerations
  • Legitimate remote administration tools (SCCM, Ansible, PDQ Deploy, SolarWinds) perform identical operations
  • IT operations and patch management workflows generate similar Sysmon events
  • Authorized penetration testing and red team exercises
  • Helpdesk remote assistance sessions
Recommended Hardening (Prevention & Impact Reduction)
  • Implement network segmentation to prevent workstation-to-workstation SMB/RPC communication
  • Disable SMBv1 across the environment; enforce SMB signing to prevent relay attacks
  • Restrict WinRM access to authorized management hosts using Windows Firewall rules
  • Deploy LAPS to eliminate shared local administrator passwords across workstations
  • Use privileged access workstations (PAW) to limit where Domain Admin credentials are used

6. Recommended Response Actions

  • Isolate the affected host from the network immediately if active exploitation or lateral movement is confirmed.
  • Identify the initiating account (from Sysmon Event 1 User field or Windows Security 4624 correlation) and disable it pending investigation.
  • Preserve Sysmon event logs from the affected host(s) before any remediation — export the Microsoft-Windows-Sysmon/Operational channel.
  • Correlate the Sysmon events with Windows Security logs using the LogonId field to link process activity to authenticated sessions.
  • Identify and remove the persistence mechanism (registry key, scheduled task, service, WMI subscription) from all affected hosts.
  • Search for the same persistence IOC (file hash, registry key path and value, task name) across all endpoints in the fleet.
  • Review the affected account's full Sysmon event history for the past 30 days to understand the full scope of attacker activity.
  • Rebuild the affected host from a known-good image if multiple persistence mechanisms or deep kernel-level persistence is suspected.
