Short Description An adversary establishes a persistent foothold that survives reboots and credential resets by exploiting Event Triggered Execution: AppInit DLLs. Windows Security Event 4657 (A registry value was modified) provides the primary detection signal when properly audited and monitored in a SIEM.

2. MITRE ATT&CK Mapping

  • Tactic : Persistence

Technique : Event Triggered Execution: AppInit DLLs (T1546.010)

3. How the Attack Typically Works (Procedure)

  1. Attacker achieves code execution on the target system with sufficient privileges to write to the target registry key.
  2. Attacker modifies or creates a registry value in a persistence location (Run keys, AppInit_DLLs, IFEO, LSA Security Packages) to point to a malicious payload.
  3. Windows Security Event 4657 is generated when the registry value is written, capturing the old and new values and the modifying process.
  4. On next trigger (logon, process start, system boot), the malicious value is evaluated and the payload executes in the appropriate context.
  5. If AppInit_DLLs or LSA packages are abused, the payload loads into every process or into the LSA process respectively, enabling credential interception.
Most common tools observed in 2024–2026:
  • Reg.exe (native Windows tool)
  • PowerShell Set-ItemProperty
  • CobaltStrike registry persistence
  • Metasploit persistence modules
  • Custom malware droppers

5. Detection / Hunting Recommendations

Strong / Classic Signature Rule (High Confidence)
  • Alert on Event ID 4657 matching the highest-fidelity field combination shown above
  • Apply thresholds appropriate to the environment: a single event may be sufficient for critical-level detections; volume thresholds apply for medium/low
  • Correlate with Event 4624 (successful logon) using SubjectLogonId to establish the session context and identify the source IP and workstation
Highest-Fidelity Correlation (SIEM Query Logic): 
EventID = 4657
AND SubjectUserName NOT IN <approved_admin_accounts>
AND (NewValue CONTAINS "powershell" OR TaskContent CONTAINS "-EncodedCommand")
Contextual / Behavioral Hunting Queries (SIEM / EDR)
  • Correlate 4657 events where SubjectLogonId links to a Type 3 (network) or Type 10 (remote interactive) logon from an external IP address
  • Alert on 4657 events from accounts in privileged groups (Domain Admins, Enterprise Admins) outside approved maintenance windows
  • Hunt for 4657 events followed within 60 seconds by lateral movement indicators (5140, 4624 to other hosts) from the same SubjectLogonId
False Positive Considerations
  • Legitimate software installers write to Run keys during installation (browsers, security tools, enterprise agents)
  • Group Policy preferences can deploy registry configurations that appear similar to persistence
  • Remote management platforms (SCCM, Ansible) may modify registry keys as part of configuration management
Recommended Hardening (Prevention & Impact Reduction)
  • Monitor all writes to HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Run and HKCU equivalent
  • Use AppLocker or WDAC to restrict which processes can write to sensitive registry paths
  • Baseline registry Run keys on all endpoints and alert on additions not matching the approved software list
  • Enable audit Object Access for registry keys via Advanced Audit Policy

6. Recommended Response Actions

  • Isolate the affected host from the network immediately if active exploitation or lateral movement is confirmed.
  • Identify the initiating account (SubjectUserName/SubjectLogonId) and disable it pending investigation.
  • Preserve all relevant Windows Security Event logs from the affected host(s) before any remediation.
  • Correlate the triggering event with Event 4624 logon records to identify the source IP, workstation, and session that performed the action.
  • Identify and remove the persistence mechanism (registry key, scheduled task, service, or DLL) from all affected hosts.
  • Search for the same persistence IOC (file hash, registry key value, task content) across all endpoints in the environment.
  • Review the affected account's full activity history for the past 30 days to understand the scope of the compromise.
  • Rebuild the affected host from a known-good image if deep persistence (bootkit, firmware, or multiple mechanisms) is suspected.
