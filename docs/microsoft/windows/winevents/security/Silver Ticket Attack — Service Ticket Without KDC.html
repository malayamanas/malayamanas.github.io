Short Description A Silver Ticket is a forged Kerberos service ticket (TGS — Ticket-Granting Service) created using the NT hash of a target service account (e.g., a computer account or service principal). Unlike a Golden Ticket, it does not require the KRBTGT hash and is scoped to a single service / host. The attacker bypasses the KDC entirely — no legitimate TGT or TGS request is sent to the domain controller.

3. How the Attack Typically Works (Procedure)

  1. Attacker obtains the NT hash of a target service account (most commonly a computer account — e.g., via DCSync, password spraying, Kerberoasting crack, or previous compromise of the machine).
  2. Using tools like Mimikatz or Rubeus, the attacker forges a TGS for a specific SPN (e.g., HOST/targetmachine, CIFS/targetmachine) with arbitrary user SID/PAC (often Domain Admin or SYSTEM privileges).
  3. The forged ticket is injected into the attacker’s session (Mimikatz kerberos::golden /target:... /service:... /rc4:... /ptt or Rubeus silver).
  4. Attacker uses the silver ticket to authenticate directly to the target service (SMB, WinRM, LDAP, etc.) — no communication with the KDC occurs for ticket issuance.
  5. Because the ticket is presented directly to the service, the target machine validates it using its own machine account key — no 4768/4769 events are generated on the domain controller.
Most common tools observed performing Silver Ticket attacks in 2024–2026:
  • Mimikatz (kerberos::golden /target:dc01.lab.local /service:cifs /rc4:... /user:Administrator /sid:... /ptt)
  • Rubeus (silver /target:... /service:host /rc4:... /user:... /ptt)
  • Impacket (some pass-the-ticket modules)
  • Custom .NET / PowerShell silver ticket implementations

5. Detection / Hunting Recommendations

Strong / Classic Signature Rule (Critical Confidence)
  • Event ID = 4624
  • LogonType = 3
  • AuthenticationPackageName = Kerberos
  • KeyLength = 0
  • No associated 4768 or 4769 in recent time for the same LogonGuid / IpAddress / WorkstationName
Behavioral / Correlation-based Hunting Queries (SIEM / EDR)
  • Kerberos Logon Type 3 with KeyLength=0 and no prior 4768/4769 in session
  • 4624 (Kerberos) + 4672 (special privileges assigned) in same LogonId — forged PAC often grants extra rights
  • Unusual privileged network logons to specific hosts (e.g., CIFS to file servers, HOST/ to RDP/WinRM targets) from non-admin IPs
  • Correlate with preceding credential access events (4662/4769 on service accounts, DCSync indicators)
False Positive Considerations
  • Rare legitimate Kerberos pass-through scenarios (constrained delegation misconfigurations)
  • Some federation / SSO bridges with direct ticket presentation
  • RODC / cached credentials in disconnected environments
Recommended Hardening (Prevention & Impact Reduction)
  • Rotate machine account passwords frequently (default 30 days → shorten to 1–7 days via GPO)
  • Monitor for NT hash exposure of computer accounts (Kerberoasting, DCSync on $ accounts)
  • Restrict high-privilege group membership in PAC via Protected Users or authentication policies
  • Enable Kerberos Armoring (FAST) and channel binding where supported
  • Use Credential Guard / LSA protection on critical servers to prevent hash extraction

6. Recommended Response Actions

  • Isolate the target machine (victim of silver ticket usage) and source workstation (IpAddress)
  • Reset machine account password of the compromised service/computer (Reset-ComputerMachinePassword or nltest /sc_reset)
  • Reset passwords of any impersonated privileged accounts appearing in logons
  • Hunt for preceding hash theft (Kerberoasting 4769 RC4, DCSync 4662, LSASS access 4656/4663)
  • Review all 4624/5140/5145 events from the source IP for lateral movement scope
  • Consider KRBTGT rotation if silver ticket was combined with golden ticket indicators
